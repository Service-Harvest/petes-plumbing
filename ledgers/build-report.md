## Post-launch fix #2: root-relative paths broke under the GitHub Pages fallback subpath URL
**Root cause**: the entire site was built with root-absolute internal paths (`href="/services"`, `href="/assets/css/main.css"`, `src="/assets/img/..."`), which is correct for a real custom domain serving from its own root. But `hexorasystems.com` (the intake's test domain) was never actually verified with GitHub Pages, so the live site was being served from GitHub's fallback project-subpath URL (`https://service-harvest.github.io/petes-plumbing/`) instead. Root-absolute paths resolve against the *domain* root under that fallback, not the *project* root where the files live — so every nav link, footer link, in-body link, the stylesheet, the favicon, and every image 404'd sitewide. This was not a Phase 4a/Phase 7 authoring defect (the CSS was fully written, the images all existed at the right paths) — it was a structural assumption that only breaks under this specific fallback hosting mode.

**Fix applied**: converted every internal `href`/`src` sitewide to depth-aware relative paths (`./` on the homepage/404.html, `../` on every other page, since all 53 category/service/hub/about/contact pages live exactly one directory below site root) — these resolve correctly under *either* a real custom domain at root *or* a GitHub Pages project subpath, so this isn't just a one-off patch. Canonical tags, OG/Twitter meta tags, and JSON-LD schema URLs were deliberately left fully absolute (`https://hexorasystems.com/...`), since those should always declare the page's real identity regardless of which URL it's being viewed through.

**Validator fix required too**: `scripts/validate.js` itself wasn't fully relative-path-aware — one check (orphan-page tracking) already stripped a leading `./`/`../` before comparing, but two others (the broken-link check and the anchor-ledger required-link check) didn't, so converting the site's paths caused ~1,400 false-positive failures in the validator until fixed. Replaced the ad-hoc regex stripping with a proper `resolveHref()` helper (using real URL-resolution semantics via Node's `URL` class) and applied it consistently everywhere an href is compared against a page URL.

**Template fix**: updated `pipeline/phase-04-repo-scaffold.md` (new Task 2, Layer 1 requirement), `pipeline/phase-06-content-drafting.md`, and all three `/snippets/` reference files to document and use this relative-path (`{{ROOT}}`) convention as the standard going forward, so future client builds don't hit the same bug when previewed via the GitHub Pages fallback URL before DNS is cut over.

**Verified**: re-ran `scripts/validate.js` (clean, 0 failures) and built a local test server that mimics the exact GitHub Pages subpath structure (`/petes-plumbing/` prefix) — confirmed in-browser that the homepage renders fully styled, the hero image loads, the chat widget loads, and clicking through to another page (Services) works correctly.

## Post-launch fix #3 (two bugs, one deploy): distorted images sitewide + popup auto-triggering again

**Bug A — every one of the 119 images was distorted (stretched/squished).**
Root cause: Gemini's `gemini-2.5-flash-image` model returns its own native output size (a 1024×1024 square) regardless of the prompt or the width/height requested — it does not honor an arbitrary target aspect ratio. The original `scripts/generate-image.js` resized with `cwebp -resize <w> <h>`, which forces the source directly to the exact target width AND height — a non-uniform stretch. Confirmed empirically: generated a test image of a plain red circle, ran it through the original resize command, and it came out as a squashed ellipse — proof the distortion was in the resize step, not a one-off content issue. Every image on the site had this same defect, since every image went through the same resize call regardless of its content or page.

Every declared `width`/`height` attribute in the HTML *did* exactly match its file's real pixel dimensions (verified sitewide, zero mismatches) — that check alone wasn't enough to catch this, since the bug was inside the file's pixel content, not in a mismatch between the file and its HTML attributes.

Fix: rewrote the resize step to scale-to-cover (uniform scale so the image fully covers the target box, preserving aspect ratio) then center-crop to the exact target size — the same effect as CSS `object-fit: cover`, baked into the file. Re-tested the same red-circle case after the fix: it stayed a circle. Regenerated all 119 images in place (same filenames, using each image's existing alt text as the regeneration prompt, since alt text already accurately described what was generated) — zero failures. No HTML changes were needed, only the image files themselves.

Updated `pipeline/phase-07-images.md` with an explicit warning about this exact pitfall (Gemini's native size vs. requested dimensions, stretch vs. cover-then-crop) so future client builds don't reintroduce it. Note: the corrected `scripts/generate-image.js` lives in this client repo only — it was never copied into `pipeline-template/scripts/`, so a fresh client build would need to re-derive this fix (or have it copied over) unless that's done separately.

**Bug B — the free-estimate popup was back to auto-showing on every page load**, the opposite of the previous fix's intent. Root cause: the embed's `data-trigger-type="click"` / `data-trigger-value=".request-estimate-cta"` configuration (set in the prior fix) is **not actually respected by GHL's `form_embed.js` script** — confirmed live: inspected the deployed page's DOM and found those exact attribute values correctly present and unchanged on the iframe element, yet the widget still auto-created and displayed its wrapper/overlay on page load regardless. In other words, the config was set correctly; the third-party script simply doesn't honor it. This means the previous fix's assumption (that `data-trigger-type="click"` is a supported, working value) was wrong, and there's no way to verify that class of undocumented third-party config against real documentation — the only way to know is to test the actual behavior, which is what caught this.

Fix: stopped relying on the embed's own trigger system entirely. Added a small inline script (present sitewide, alongside the embed) that actively suppresses the widget's own wrapper/overlay elements (`#popup-xHJhkUk8Wps0ag0Dokml-wrapper` and `-overlay`, both identified by inspecting the live DOM while the popup was open) via a `MutationObserver` plus timed sweeps, immediately hiding either one the instant the third-party script creates or shows them — and only removes that suppression when a genuine click on `.request-estimate-cta` is detected, at which point the popup is allowed to show and stays allowed for the rest of that page view. This does not depend on the third-party script's own trigger semantics at all; it directly controls the DOM elements the script creates.

**Follow-up found during live verification**: after deploying the suppression script, confirmed on the live site that auto-show was fixed, but clicking "Request a Free Estimate" also did nothing visible — the widget's own `.ep-overlay` CSS class sets `display: none` as its *default* (confirmed by inspecting the live page's injected stylesheet), so removing our suppression override just fell back to that same hidden default instead of actually showing it. Fixed by having the click handler explicitly set `display: flex` on the overlay and `display: block` on the wrapper (matching what the widget itself sets when it auto-shows) rather than merely removing our override. Verified this exact fix directly in the live page's console before updating the file, then confirmed the corrected script produces the same result end-to-end after redeploying.

**Final end-to-end live verification** (commit `e6803a3`, deployed and confirmed green): loaded the homepage fresh (no cache) — no popup auto-opened, hero image rendered fully proportional (no stretch/squish). Clicked "Request a Free Estimate" — the full lead-capture form (Full Name, Email, Phone, Address, City, State fields) rendered correctly inside the popup. Repeated on the Services hub page (a non-homepage page, reached via its real relative path) — same result: no auto-popup on load, hero image proportional, confirming both fixes hold sitewide and not just on the page originally used to diagnose them. Re-ran `scripts/validate.js`: clean, 0 failures. Both bugs are closed.

## Post-launch fix #4: design-system modernization (spacing, container width, typography, color/depth/icons)
Client feedback: the live site looked plain/dated, with noticeably wide desktop side margins. Diagnosed `main.css` directly (single flat 1140px container reused everywhere, one flat alt-background stripe, single-weight system-font headings, zero box-shadows, no hover motion) before changing anything, and reported the diagnostic back for sign-off. Root cause was judged to be mostly `phase-04a-design-system.md`'s Task 1 quality bar itself ("consistent and WCAG AA accessible" is a floor, not a modern-feel target) rather than this client's thin intake — the intake did ask for "modern, polished, not generic," but the phase file never translated that into a concrete bar for spacing generosity, depth, type pairing, or full-bleed section rhythm, so even a client with strong brand direction would have gotten a similarly plain result.

Implemented all four proposed areas in `site/assets/css/main.css`:
- **Spacing**: added `--space-6` (a `clamp(3.5rem, 6vw, 5.5rem)` tier) for section vertical padding, replacing the flatter `--space-4` (2.5rem); `clamp()`-based container/header/footer side padding so it scales on ultra-wide screens.
- **Container/full-bleed**: `--container-max` raised from 1140px to 1280px; body copy (`p`) capped at a separate 68ch measure independent of container width for readability; section background colors already applied to the full-width `<section>` element (not just the inner `.container`), so alternating section colors now read as full-bleed rhythm rather than a single flat stripe.
